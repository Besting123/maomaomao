# 喵伴云养 —— UI 设计课程作业版计划

> 本文档按最新定位重写：本项目当前首先是 **用户界面设计课程作业 / 高保真 Android Compose 前端原型**。近期目标不是做完整线上产品，而是把界面、交互、视觉叙事、地图与陪伴核心体验打磨到可展示、可测试、可讲清设计逻辑。后端、数据库、账号、真实数据同步与管理员流程放到后续产品化阶段。

---

## 1. 课程作业定位

### 1.1 当前交付物是什么

本阶段交付物是一个可运行的 Android 前端高保真原型，重点证明：

- 信息架构清楚：用户知道从哪里进入地图、陪伴、学堂、任务、论坛、个人页。
- 视觉系统完整：颜色、卡片、按钮、导航、图标、间距、圆角、层级统一。
- 关键交互可信：按钮有反馈，页面之间能跳转，核心路径能顺畅演示。
- 特色场景突出：校园猫地图与 3D 云陪伴是作品记忆点。
- 动物福利表达明确：界面不鼓励追猫、围观、过度投喂或暴露敏感位置。

### 1.2 当前不作为主要交付物的内容

以下内容后置，不影响课程作业前端验收：

- 真实账号、登录注册、权限系统；
- Room / DataStore / 云数据库；
- Retrofit / Firebase / Supabase 等远程接口；
- 论坛真实发帖、审核、评论、通知；
- 志愿者后台、管理员审核、线下任务派发；
- BJTU 猫谱 / 红果园猫猫队真实数据同步；
- 真实投喂点、精确位置、维护路线公开展示。

## 2. 重新调研后的设计依据

### 2.1 UI 课程与高保真原型方法

- Coursera Google UX 高保真原型课程强调：高保真原型应“像成品一样可操作”，并通过可用性测试收集反馈、迭代设计。参考：https://www.coursera.org/programs/youngster-co-google-learning-program-a46xf/learn/high-fidelity-designs-prototype
- Udacity Mobile UX for Developers 强调用户中心设计、移动端约束与 Android 体验打磨。参考：https://www.udacity.com/course/ux-design-for-mobile-developers--ud849
- IxDF Mobile UI Design 强调小屏幕上的视觉层级、触控目标、导航效率与减少认知摩擦。参考：https://www.interaction-design.org/courses/mobile-ui-design-course

对本项目的启发：课程作业阶段应优先完成 **高保真界面、核心路径、交互反馈、设计说明与可用性检查**，不应把主要精力放在后端工程。

### 2.2 虚拟宠物 / 云陪伴产品参考

- Neko Atsume 的优势是低压力、低文字、低门槛：用户放置物品后等待猫咪出现，依靠可爱反馈和收集感形成留存。参考：https://www.nekoatsume.com/en/ 与 https://medium.com/@laurielepham/neko-atsume-a-lesson-in-universal-usability-4c54a1507a84
- Tamagotchi / My Talking Tom 类产品常见闭环是：状态需求 → 互动照顾 → 即时反馈 → 奖励或成长。参考：https://mytalkingtom.com/gameplay
- 虚拟宠物界面通常包含：中心宠物形象、少量状态 HUD、短动作反馈、日常任务、奖励物品、轻量养成。

对本项目的启发：云陪伴页不必做复杂后端，也能通过 **3D 猫、状态条、动作按钮、反馈气泡、陪伴记录、小鱼干奖励** 做出完整前端体验。

### 2.3 校园猫地图 / 动物福利项目参考

- Georgia Tech Campus Cats app 支持猫咪照片、档案、目击地图和目击上报；但投喂站只对组织成员可见，以保护猫咪安全并避免弃养。参考：https://research.gatech.edu/find-feline-new-app-helps-georgia-tech-track-its-campus-cats
- Georgia Tech Campus Cats 网站允许上传猫咪目击信息，包括照片、地点、日期、时间和备注。参考：https://www.gtcampuscats.org/sightings
- iNaturalist 的地理隐私机制提供了“公开 / 模糊 / 私密”位置表达思路。参考：https://help.inaturalist.org/en/support/solutions/articles/151000169938
- Neko Map / Meow Mapper 等校园猫项目通常包含猫咪档案、地图、志愿者、喂养或照护状态。参考：https://neko-map.github.io/ 与 https://meow-mapper.github.io/

对本项目的启发：地图页应优先做成 **安全引导型地图 UI**，不是“精准找猫工具”。公开界面展示区域级热点和安全建议即可；精确点位、投喂站和志愿者路线放到后续权限阶段。

## 3. 当前代码状态判断

### 3.1 已具备的前端原型能力

- **基础框架**：单 Activity、Jetpack Compose、Material 3、底部五栏导航、二级页面路由已完成。
- **应用品牌**：`app_name` 已是“喵伴云养”。
- **核心页面**：首页、校园、陪伴、论坛、我的、猫咪档案、任务中心、新手学堂均已存在。
- **本地状态**：`MainViewModel` 使用 `StateFlow` 模拟小鱼干、签到、任务、答题、陪伴记录、等级、经验、饱食、心情、健康。
- **3D 猫**：`CatModel3DViewer` 已加载 `assets/models/cat.glb` 与 `assets/environments/studio.hdr`，支持拖拽惯性旋转和双击回调。
- **校园地图**：`CampusScreen` 已使用 osmdroid + 高德地图瓦片，展示 BJTU 区域、猫咪 marker、建筑 marker 和关怀路线。
- **前端模拟交互**：任务、答题、论坛报名/响应、个人页兑换等已有本地反馈。

### 3.2 需要重新表述的地方

- 当前是 **高保真 UI 原型**，不是完整产品。
- 地图 marker 与底部卡片尚未完全联动：`selectedHotspot` 主要仍是本地默认状态。
- `CampusScreen` 中热点数据与地图 marker 数据仍有重复，后续应统一为一个 UI 数据源。
- `CatModel3DViewer` 接收 `animationIndex`，但目前没有真正驱动 GLB 动作切换；陪伴动作主要靠状态条、粒子、气泡和记录体现。
- 论坛分类、发布、评论、求助等是前端模拟，不是真实社区系统。
- Profile / CatProfile 中部分记录仍是静态展示，跨页面数据互通只完成了一部分。

## 4. 新开发原则

1. **UI 课程优先**：优先打磨界面完成度、视觉一致性、交互反馈和演示路径。
2. **高保真而非全功能**：允许使用 mock 数据，但界面要像真实产品一样可操作。
3. **先做好少量核心体验**：一只示范猫、一条演示路径、一张校园地图先做到精致。
4. **数据后置但结构清楚**：样例数据可以硬编码或放在 ViewModel，但命名要便于后续替换为真实数据源。
5. **地图保护猫咪安全**：公开页面只展示区域级信息；精确位置和投喂站后续做权限控制。
6. **陪伴弱化投喂冲动**：交互重点从“喂”转向观察、补水、安抚、记录、学习。
7. **所有可点击元素必须有反馈**：Toast、Dialog、状态切换、气泡、动画或页面跳转均可。
8. **文档要服务展示**：计划、验收和演示脚本应能帮助课程答辩说明设计逻辑。

## 5. 阶段路线图

### P0：视觉系统与作品完整度

**目标：** 让 App 看起来像完整课程作品，而不是拼接页面。

任务：

- 统一首页、校园、陪伴、论坛、我的、任务、新手学堂、猫咪档案的卡片、间距、圆角、按钮、图标与顶部栏。
- 检查页面底部留白，避免底部导航遮挡关键操作。
- 清理“演示功能”“前端模拟”这类直接暴露给普通用户的文案，必要时改成自然产品文案。
- 为主要页面准备截图和说明：页面目标、用户任务、设计亮点。

验收：

- 主要页面视觉风格统一。
- 课程展示时不会出现明显模板感或开发占位感。
- 每个主页面都能说明它在用户旅程中的作用。

---

### P1：核心演示路径与信息架构

**目标：** 形成一条可演示、可讲解的用户路径。

推荐路径：

1. 首页查看今日状态与入口；
2. 任务中心签到 / 完成任务获得小鱼干；
3. 新手学堂学习正确接近猫咪并答题；
4. 校园地图查看区域热点与安全标签；
5. 猫咪档案了解个性、边界和健康建议；
6. 云陪伴进行观察、安抚、补水等互动；
7. 我的页面查看参与痕迹和善意记录。

任务：

- 确保首页、地图、档案、陪伴、任务、学堂之间的跳转路径清楚。
- 给“分享、更多、通知、评论、发帖、报名、兑换”等操作补齐自然反馈。
- 用本地状态模拟奖励、完成、已报名、已收藏、已响应等变化。

验收：

- 不接后端也能完整演示一条用户旅程。
- 用户点击关键按钮后能明确知道发生了什么。

---

### P2：两大特色界面重点打磨

#### P2-A 校园地图界面

**目标：** 地图成为“安全接近猫咪”的 UI 展示场景。

优化方向：

- 统一 `CampusHotspot` UI 数据源，同时驱动 marker、底部卡片、常驻猫咪和路线。
- 点击 marker 后更新底部卡片并展开详情。
- 时间段切换不仅改变文案，也改变推荐热点、路线强调或安全提示。
- 底部卡片展示：照片、区域名、状态标签、最近记录、接近指南、查看档案。
- 公开界面使用区域级描述，不展示真实投喂站和精确维护路线。
- 后续志愿者模式再考虑显示精确点位、任务清单、完成凭证。
- 补充 osmdroid `MapView` 生命周期处理，避免资源泄漏。

课程验收重点：

- 地图视觉有 BJTU 校园氛围。
- 用户能理解“哪里适合远观，哪里不要打扰”。
- 地图自然导向猫咪档案和云陪伴。

#### P2-B 云陪伴界面

**目标：** 陪伴页成为项目最有记忆点的情感化界面。

优化方向：

- 保持“3D 猫居中 + HUD + 操作栏 + 气泡反馈”的结构。
- 状态 HUD 建议聚焦：饱食、心情、健康、安全感 / 精力。
- 互动优先级调整为：观察、补水、安抚、记录；添粮应带安全限制提示。
- 修复余额不足时仍显示成功反馈的问题。
- 如果 GLB 有动画，后续让 `animationIndex` 真正触发动作；如果没有，则把动作反馈明确放在粒子、表情文案和状态变化上。
- 增加低压力留存感：陪伴后生成照片、心情、足迹或小故事片段。

课程验收重点：

- 用户操作后有即时视觉反馈。
- 猫咪状态变化易懂。
- “陪伴”概念强于“投喂”。

### P3：次级页面前端模拟闭环

**目标：** 让任务、学堂、论坛、我的、档案看起来完整可用。

任务中心：

- 签到、任务完成、奖励领取、已完成状态继续使用 ViewModel 模拟。
- 明确区分“学习任务、观察任务、陪伴任务”。

新手学堂：

- 课程卡片、详情弹窗、答题反馈、奖励动画继续完善。
- 内容重点围绕“观察猫咪信号、保持距离、补水优先、避免错误投喂”。

论坛：

- 分类 chip 应真正筛选本地样例帖子。
- 发帖弹窗可以先本地新增一条帖子或展示提交成功状态。
- 报名、响应、收藏、点赞继续本地模拟。

我的页面：

- 展示用户参与痕迹：小鱼干、任务完成、学习进度、陪伴记录、关注猫咪。
- 不再出现“后端未接入”式表达，改为自然产品状态。

猫咪档案：

- 继续做单猫高质量示范，不追求覆盖所有猫。
- 重点展示：个性、互动边界、健康建议、出现规律、陪伴记录、3D 形象。

验收：

- 主要次级页面都有可点击反馈。
- 不依赖真实后端也能呈现完整界面逻辑。

---

### P4：课程展示与可用性验证

**目标：** 为课程汇报准备设计证据，而不仅是代码功能。

交付内容：

- 1 条 3-5 分钟演示脚本；
- 8-10 张关键界面截图；
- 1 页用户旅程图；
- 1 页视觉系统说明：颜色、字体、组件、图标风格；
- 1 页安全与伦理说明：为什么不公开精确投喂点；
- 简短可用性测试记录：让 2-3 名同学按演示路径操作，记录卡点。

测试问题：

- 用户是否知道下一步该点哪里？
- 用户是否理解小鱼干、任务、地图、陪伴之间的关系？
- 用户是否意识到“不要随意追猫/喂猫”？
- 地图与陪伴是否是最有记忆点的两个界面？

---

### P5：后续数据互通与产品化阶段

**目标：** 前端作品完成后，再逐步接入真实数据与后端能力。

后续内容：

- 抽出 UI 数据模型：Cat、CampusHotspot、CompanionRecord、Task、ForumPost、UserProfile。
- 增加 Repository 层，替换硬编码和 ViewModel 内部样例数据。
- 本地持久化：DataStore / Room。
- 远程同步：API、账号、权限、论坛、审核、通知。
- 真实 BJTU 猫数据接入前，确认授权、图片版权、平台规则、学生隐私和动物安全。
- 志愿者 / 管理员权限：精确投喂点、维护路线、审核后台、紧急响应。

验收：

- UI 层不需要大改即可替换数据源。
- 真实点位和敏感数据有权限与模糊展示策略。

## 7. 技术策略

### 当前技术栈

- 平台：Android / Kotlin
- UI：Jetpack Compose + Material 3
- 导航：Navigation Compose
- 状态：ViewModel + StateFlow
- 3D：SceneView / Filament，`assets/models/cat.glb`，`assets/environments/studio.hdr`
- 地图：osmdroid + 高德地图瓦片
- 数据：ViewModel 内存态 + 静态样例资源

### 近期策略

- 不新增后端依赖。
- 不引入真实账号或数据库。
- 优先整理 UI 数据结构与页面状态。
- 使用本地 mock 数据完成交互演示。
- 用 Gradle build/test 验证工程可运行。
