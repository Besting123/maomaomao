# 喵伴云养 —— 开发计划与纠偏策略

> 本文档基于 `goal.md`、当前 Android Compose 代码与需求差距分析更新。当前项目已经具备较完整的高保真 UI 原型，但仍不是完整产品。根据最新开发策略，近期优先级调整为：**前端 UI 设计完成度、3D/动态猫咪形象、校园地图体验**；后端、数据库、真实数据同步与生产级业务闭环后续补充。

---

## 1. 当前项目状态判断

### 1.1 已完成的内容

- **Android Compose 基础框架**：单 Activity、Material 3、底部五栏导航、二级页面路由已完成。
- **核心页面 UI**：首页、校园、陪伴、论坛、我的、猫咪档案、任务中心、新手学堂均已有较完整界面。
- **少量内存态交互**：`MainViewModel` 已支持小鱼干余额、签到、任务完成、答题奖励、陪伴互动记录。
- **陪伴叙事初步建立**：导航已从“投喂”转为“陪伴”，页面文案也开始强调科学互动、补水优先、互动边界。
- **视觉素材与主题**：已有大量猫咪/校园图片资源，色彩系统基本统一。

### 1.2 当前本质

当前项目是 **UI-first 的 Android Compose 概念原型**。它能展示“喵伴云养”的主要体验路径，但不能被视为功能完整产品。

当前真实功能主要局限在：

- 进程内 `StateFlow` 状态；
- 硬编码任务、课程、猫咪、帖子、地图点位；
- 点击按钮后局部 UI 变化或内存数据变化。

尚未具备：

- 本地持久化；
- 真实 BJTU 猫谱/红果园猫猫队数据接入；
- 真实地图或可维护的校园区域数据；
- 3D 猫咪模型或高质量动态虚拟形象；
- 论坛、组队、线下认领的真实业务流；
- 数据安全、动物福利与位置暴露策略。

## 3. 需要纠正的思路

### 3.1 不能把“UI 完成”误判为“产品完成”

当前页面覆盖度很高，但多数数据是硬编码或内存态。近期不优先补后端，而是先把 UI、3D/动态猫咪、校园地图做成完整演示；同时用前端 mock 状态减少页面割裂，避免继续只堆静态页面。

### 3.2 地图不能只做“找到猫”

项目目标强调“学会正确接近猫”。因此地图应承担教育与风险控制职责：

- 不鼓励用户追逐猫；
- 不暴露敏感精确位置；
- 公众侧默认使用粗粒度区域、延迟时间与可信度标签；
- 精确点位只允许在未来志愿者/管理员角色中按需开放；
- 对休息、病弱、应激、幼猫等场景给出“不打扰”提示；
- 地点信息应使用区域级、延迟或模糊展示。

### 3.3 BJTU 猫谱数据不能默认可用

微信小程序、社群图片、校园照片可能涉及版权、平台规则、志愿者授权、学生隐私与动物安全。正式接入前必须确认合法来源。未确认前，产品只使用样例数据或已获许可数据。

### 3.4 3D 猫不是普通 UI 组件

“3D 建模特效”涉及模型资产、授权、动画、性能、APK 体积和 Compose 集成。应作为可行性验证，不应直接承诺完整 3D 系统。

可选路线：

- **实现方案**：SceneView / Filament，加载 `glTF/GLB` 模型，模型放入 `app/src/main/assets/models/`。
- **资产要求**：猫咪模型必须使用真实 3D 资产，推荐 `cat.glb`，内嵌贴图与基础动画。
- **验收要求**：至少一只猫具备可触发动作、反馈文案和状态变化。

### 3.5 后端不是当前优先项

现阶段更适合先把前端展示做完整：UI、3D/动态猫咪、校园地图、演示交互。后端、账号、数据库、同步、审核、管理员工具应作为后续产品化阶段；近期只需保留清晰的数据接口意识，避免把前端写死到无法接入后端。

## 4. 新开发原则

1. **前端展示优先**：近期目标是把 UI、3D/动态形象、校园地图做到可演示、可信、好看。
2. **后端后置**：数据库、账号、同步、审核、真实接口暂不作为当前阶段目标。
3. **允许 mock 数据**：前端阶段可继续使用本地样例数据、ViewModel 状态或静态资源，但命名和结构要便于后续替换成后端数据。
4. **一只猫先做到精致**：优先完成一只示范猫的动态形象、档案、互动反馈和陪伴感。
5. **地图是重点场景**：校园地图需要在视觉、交互、区域信息、接近指南上优先打磨。
6. **所有按钮必须有前端反馈**：即使后端未实现，也不能出现点了没反应的空 `onClick`。
7. **动物福利优先**：地图和互动文案仍要避免鼓励围观、追猫、过度投喂或暴露敏感位置。

## 5. 新阶段计划

### P0：前端 UI 完整度与演示可信度

**目标：** 先把现有 App 做成完整、连贯、可展示的前端作品。

#### 5.1 品牌与基础视觉

- 将 `app_name` 从 `My Application` 改为 `喵伴云养`。
- 统一首页、校园、陪伴、论坛、我的、猫咪档案、新手学堂、任务中心的视觉层级。
- 统一卡片圆角、间距、图标风格、按钮样式、顶部栏和底部导航样式。
- 检查深浅色、状态栏、导航栏、页面底部留白和滚动体验。

#### 5.2 空操作前端反馈

- 通知、分享、更多、兑换奖励、认领提醒、发帖、评论、报名、求助响应等按钮先补前端反馈。
- 可采用 Toast / Snackbar / Dialog / 页面内状态变化，提示“演示功能”“即将开放”或模拟状态改变。
- 保留演示可信度，不出现点击无反馈。

#### 5.3 首页与核心路径强化

- 首页突出三条主路径：进入校园地图、进入云陪伴、进入新手学堂。
- “我关注的猫咪”“今日猫咪动态”“任务提醒”先使用样例数据展示完整状态。
- 准备一条完整演示路径：签到 → 学习 → 地图查看 → 进入猫咪档案 → 云陪伴互动。

**验收标准：**

- 主要页面视觉统一，没有明显模板默认信息。
- 主要按钮都有反馈。
- 不接后端也能完成一条顺畅演示路径。

---

### P1：3D/动态猫咪形象优先攻关

**目标：** 做出项目最有记忆点的“猫咪陪伴形象”。

#### 5.4 3D 方案验证

- 优先验证 SceneView / Filament 方案。
- 模型格式优先采用 `GLB` / `glTF`。
- 模型资源放置在 `app/src/main/assets/models/`。
- 验证模型加载、自动动画、旋转/触摸、性能与 APK 体积。

#### 5.5 3D 模型资产制作

- 使用 Blender、Maya、C4D 或其他建模软件制作猫咪模型。
- 导出为 `GLB` 或 `glTF`，优先使用单文件 `cat.glb`。
- 模型应包含基础材质、贴图和可自动播放的待机动画。
- 后续再增加被安抚、喝水、休息、警惕等动作动画。

#### 5.6 与页面联动

- 动态猫咪优先放在 `CompanionScreen` 和 `CatProfileScreen`。
- 用户点击安抚、观察、补水、添粮时，猫咪形象要有前端反馈。
- 前端可先用本地状态模拟心情、防备心、亲密度变化。

**验收标准：**

- 至少一只示范猫有真实 3D 互动形象。
- 用户互动后能看到明确的视觉反馈。
- Android Studio 可同步 SceneView 依赖，模型资源可从 `app/src/main/assets/models/cat.glb` 加载。

---

### P2：校园地图体验重点打磨

**目标：** 将校园地图做成第二个核心展示场景。

#### 5.7 地图视觉升级

- 强化 BJTU 校园感：区域、道路、建筑、绿地、猫咪活动区的视觉层次更清楚。
- 优化猫咪热点样式、区域卡片、地点标签、动线提示。
- 增加时间段切换的前端表现：清晨、午后、傍晚、夜间。

#### 5.8 地图交互升级

- 点击区域/热点后展示猫咪信息、互动建议、接近指南。
- 增加“不打扰”“适合远观”“可补水”“仅志愿者处理”等安全标签。
- 首页、猫咪档案、地图之间建立前端跳转关系。

#### 5.9 安全表达

- 地图展示以区域级为主，不做实时精确定位。
- 文案避免“快去找它”，改为“远观记录”“保持距离”“观察状态”。
- 对病弱、幼猫、休息区等场景强调不要打扰。

**验收标准：**

- 地图页面视觉上能体现 BJTU 校园环境。
- 点击热点有完整信息卡和安全建议。
- 地图能自然导向猫咪档案和云陪伴页面。

---

### P3：前端交互闭环增强

**目标：** 在不接后端的前提下，让主要功能看起来完整可用。

#### 5.10 云陪伴前端闭环

- 使用本地状态模拟小鱼干、互动记录、猫咪状态变化。
- 互动后生成陪伴记录，并在陪伴页/档案页/我的页展示一致内容。
- 余额不足、互动成功、状态变化都要有明确 UI 反馈。

#### 5.11 新手教育与任务

- 课程卡片可以进入详情或弹窗。
- 答题后展示正确/错误反馈和奖励动画。
- 任务中心先用前端状态模拟完成、领取、已领取。

#### 5.12 论坛、组队、线下联动前端模拟

- 论坛分类切换能筛选前端样例帖子。
- 发帖按钮可打开模拟发布弹窗。
- 组队报名按钮能切换“已报名/取消报名”。
- 认领提醒按钮能切换“已认领/待完成”。

**验收标准：**

- 核心交互不依赖后端即可演示。
- 页面之间的状态尽量一致。
- 空按钮全部被前端反馈替代。

---

### P4：后端与真实数据预留阶段（后续）

**目标：** 当前不实现后端，但保留后续接入空间。

- 整理数据模型命名，避免 UI 文案和数据结构完全耦合。
- 后续再补 Repository、DataStore/Room、网络接口、账号、审核、管理员工具。
- BJTU 猫谱或红果园猫猫队数据接入前，需要确认授权、图片版权、位置安全和更新方式。
- 真实地图、真实点位、志愿者流程、求助审核都放入后续产品化阶段。

**验收标准：**

- 当前代码不强依赖真实后端。
- 前端样例数据结构便于未来替换。
- 文档明确说明后端后续补充。

## 7. 技术栈与架构计划

### 当前技术栈

- **平台：** Android (Kotlin)
- **UI 框架：** Jetpack Compose + Material 3
- **导航：** Jetpack Navigation Compose
- **状态管理：** ViewModel + StateFlow
- **当前数据：** ViewModel 内存模拟数据 + 静态样例资源

### 近期技术策略

- **UI：** 继续使用 Compose 完成前端页面、动效、状态反馈。
- **3D/动效：** 使用 SceneView / Filament + GLB/GLTF 模型。
- **地图：** 当前继续 Compose Canvas / 静态图 / 自定义组件路线，优先做好视觉和交互，不急于接入地图 SDK。
- **数据：** 近期使用 mock/sample data 与 ViewModel 本地状态，后端后续补充。
- **后端：** 暂不作为当前开发目标；只保留接口意识和数据结构可替换性。
